# 小说续写 Agent — 优化落地方案（v2）

## 一、项目概述

**定位**：自用小说续写工具，项目制管理，人工把控核心规划，AI执行正文续写。强调降低维护负担、提升续写质量与一致性。

**技术栈**：
- 后端：Python 3.11+ / FastAPI
- 前端：React + TailwindCSS（Vite构建）
- 数据库：SQLite（正文与元数据统一存储，单文件备份友好）
- AI模型：DeepSeek / MiMo（可切换，1M上下文）
- 部署：本地运行，localhost访问

**核心设计理念**：
- 大纲层级精简至"章纲+场景要点"，弱化中间卷/阶段。
- 上下文采用"最近15章全文 + 全局纲要 + 按需检索片段"分层混合模式，底线保留近距离连贯性，避免长文噪声。
- 伏笔、时间线与角色状态由AI每章自动提取更新，人工仅审阅。
- 操作前端以向导式按钮为主，自然语言对话为辅，避免意图误识打断创作流。

## 三、数据模型（精简后）

### 3.1 数据库表（SQLite）

```sql
-- 项目表（增加分卷概要字段作为总纲）
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    genre TEXT,
    description TEXT,
    model_provider TEXT DEFAULT 'deepseek',
    model_name TEXT DEFAULT 'deepseek-chat',
    target_words INTEGER DEFAULT 200000,
    current_words INTEGER DEFAULT 0,
    current_chapter INTEGER DEFAULT 0,
    style_notes TEXT,                    -- 人工风格备注
    volume_summaries TEXT,              -- 分卷概要（总纲，Markdown，例如每卷200字）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 章节表（正文直接存入content字段）
CREATE TABLE chapters (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    chapter_number INTEGER NOT NULL,
    title TEXT,
    content TEXT NOT NULL,              -- 章节正文（保持原格式）
    word_count INTEGER DEFAULT 0,
    summary TEXT,                       -- AI生成摘要（备用）
    status TEXT DEFAULT 'draft',        -- draft/reviewed/final
    volume_label TEXT,                  -- 可选：所属卷标（如"第一卷"）
    arc_label TEXT,                     -- 可选：所属弧线标签
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id),
    UNIQUE(project_id, chapter_number)
);

-- 章纲表（核心大纲：每章规划）
CREATE TABLE chapter_outlines (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    chapter_number INTEGER NOT NULL,
    title TEXT,                         -- 章节标题
    core_objective TEXT,               -- 本章核心目标（推动情节的点）
    emotional_arc TEXT,                 -- 情感走向
    hooks TEXT,                         -- 章末钩子
    version INTEGER DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id),
    UNIQUE(project_id, chapter_number)
);

-- 场景要点表（细纲的轻量版，绑定到章）
CREATE TABLE scene_points (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    chapter_number INTEGER NOT NULL,
    scene_order INTEGER NOT NULL,       -- 场景序号
    mission TEXT,                      -- 场景任务（这一小段要完成什么）
    key_dialogue_hint TEXT,            -- 关键对话提示
    atmosphere TEXT,                   -- 氛围/环境描写要点
    target_words_ratio REAL DEFAULT 0.25, -- 该场景占本章字数比例建议
    FOREIGN KEY (project_id) REFERENCES projects(id),
    UNIQUE(project_id, chapter_number, scene_order)
);

-- 角色表（保留，增加状态记录将由快照表承载）
CREATE TABLE characters (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    name TEXT NOT NULL,
    role TEXT,                           -- 主角/配角等
    age TEXT,
    personality TEXT,
    speech_style TEXT,
    appearance TEXT,
    background TEXT,
    relationships TEXT,                 -- JSON
    character_arc_summary TEXT,         -- 角色弧线简述
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id, name) REFERENCES characters(project_id, name)
);

-- 角色状态快照表（每章续写后由AI生成最新状态）
CREATE TABLE character_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    chapter_number INTEGER NOT NULL,
    character_name TEXT NOT NULL,
    current_state TEXT NOT NULL,         -- 位置、情绪、关系变化、能力/物品变化等（结构化JSON或短文）
    extracted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id),
    UNIQUE(project_id, chapter_number, character_name)
);

-- 风格档案表（保留，但增加动态控制参数偏好）
CREATE TABLE style_profiles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL UNIQUE,
    base_analysis TEXT,                 -- AI初始风格分析文本
    human_notes TEXT,
    default_description_density INTEGER DEFAULT 3,  -- 描写密度1-5
    default_dialogue_ratio INTEGER DEFAULT 3,       -- 对话占比1-5
    default_pacing TEXT DEFAULT 'medium',           -- 节奏：slow/medium/fast
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id)
);

-- 伏笔与悬念表（由AI自动更新，人工确认）
CREATE TABLE foreshadowing (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    description TEXT NOT NULL,
    planted_chapter INTEGER NOT NULL,
    expected_reveal_chapter INTEGER,
    actual_reveal_chapter INTEGER,
    status TEXT DEFAULT 'active',        -- active/resolved/abandoned
    importance TEXT DEFAULT 'normal',
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id)
);

-- 故事时间线表（由AI自动更新）
CREATE TABLE timeline (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    chapter_number INTEGER NOT NULL,
    story_time_description TEXT NOT NULL, -- 如"第三天深夜"
    story_date TEXT,
    duration TEXT,
    summary TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id)
);

-- 对话历史（辅助对话）
CREATE TABLE chat_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    role TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id)
);
```

### 3.2 正文与文件存储

- 所有正文存储在 `chapters.content` 字段（SQLite TEXT），单文件备份即可携带全部数据。
- **不再使用文件系统存储章节正文**，避免同步问题。
- 如果有大量原文需要导入，前端支持粘贴或文件上传，后端存入 `source_text` 表（可独立保留）。

### 3.3 简化后的大纲结构示意

```
分卷概要 (projects.volume_summaries)   ← 总纲，500字左右
    │
    ├─ 章纲 (chapter_outlines)         ← 每章核心目标+情感+钩子
    │     └─ 场景要点 (scene_points)   ← 场景任务、对话提示、氛围
    │
    └─ 无独立卷/阶段纲，通过卷标(volume_label)和弧线标签(arc_label)柔性分组
```

## 四、API 设计（调整后）

删减了总纲/卷纲/阶段纲的独立REST接口，整合为项目属性与章纲操作。增加自动提取相关端点。

### 4.1 项目管理
不变，增加 `volume_summaries` 字段。

### 4.2 章纲与场景要点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/projects/{id}/outlines/chapters` | 获取所有章纲摘要列表 |
| GET | `/api/projects/{id}/outlines/chapters/{chapter}` | 获取某章纲详情（含场景要点） |
| PUT | `/api/projects/{id}/outlines/chapters/{chapter}` | 更新章纲 |
| POST | `/api/projects/{id}/outlines/chapters/{chapter}/generate` | AI生成某章纲+场景要点（基于分卷概要与前文） |
| POST | `/api/projects/{id}/outlines/chapters/{chapter}/scenes` | 手动添加/调整场景要点 |
| DELETE | `/api/projects/{id}/outlines/chapters/{chapter}/scenes/{order}` | 删除场景要点 |

### 4.3 章节续写（核心）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/projects/{id}/chapters/{chapter}/write` | **启动续写向导**，返回所需预览信息（当前上下文范围、章纲+场景要点预览、风格参数等），用户微调后调用正式生成 |
| POST | `/api/projects/{id}/chapters/{chapter}/generate` | 执行正式续写（流式），完成后自动触发后处理（提取伏笔/时间/角色快照） |
| GET | `/api/projects/{id}/chapters/{chapter}` | 获取章节正文 |
| PUT | `/api/projects/{id}/chapters/{chapter}` | 手动编辑章节 |

### 4.4 角色管理
不变，增加 `character_snapshots` 相关查询。

### 4.5 伏笔与时间线（自动维护为主）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/projects/{id}/foreshadowing` | 获取所有伏笔（支持过滤状态） |
| PUT | `/api/projects/{id}/foreshadowing/{fid}` | 人工修改/确认伏笔状态 |
| GET | `/api/projects/{id}/timeline` | 获取时间线 |
| POST | `/api/projects/{id}/chapters/{chapter}/extract-meta` | 手动触发对已写章节的元数据提取（伏笔/时间/快照） |

### 4.6 风格控制

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/projects/{id}/style` | 获取风格档案与当前控制参数 |
| PUT | `/api/projects/{id}/style/params` | 更新描写密度、对话占比、节奏等参数（影响后续续写） |

### 4.7 辅助对话（权责变轻）
保留对话端点，但前端主要作为"常见问题解答"和"自由指令"的入口，不承担核心操作。

### 4.8 模型管理
不变。

## 五、Prompt 设计变化

### 5.1 上下文组装策略（分层混合）

**核心原则：最近15章全文是必须的底线，其余部分按需提供摘要或检索片段。**

#### 5.1.1 上下文窗口组成

```
┌──────────────────────────────────────────────────────────────┐
│  续写第N章的Prompt上下文                                      │
├──────────────────────────────────────────────────────────────┤
│ 1. 系统指令（角色、输出格式、写作规则）                        │
│ 2. 分卷概要（总纲，500字内）                                  │
│ 3. 当前章纲 + 场景要点                                        │
│ 4. 风格控制参数（密度、对话占比、节奏，一行描述）              │
│ 5. 活跃伏笔摘要（由AI提炼，200字内，仅包含未揭示且相关的伏笔） │
│ 6. 故事时间线摘要（最近10章的时间节点，防止矛盾）              │
│ 7. 角色信息：                                │
│    a. 全部角色档案（静态属性）                  │
│    b. 最近一次角色状态快照（出场角色最新状态）   │
│ 8. 正文部分（分层注入）：                       │
│    a. 【硬底线】第N-15章 至 第N-1章 全文       │
│    b. 【检索增强】更早章节的相关片段：           │
│       - 如当前章涉及特定伏笔或角色，检索该线索最早出现的原文段落（≤500字/段）│
│       - 如当前章与某早期事件强呼应，检索那章关键段落  │
│ 9. 生成输出空间（2000-3000字正文）               │
└──────────────────────────────────────────────────────────────┘
```

- 15章底线保证了近距情节、对白风格、即时状态的完美连贯。
- 检索增强用关键词匹配（如角色名、伏笔关键词）在本地数据库文本中搜索相邻段落，无需向量库。

#### 5.1.2 对话续写 Prompt 示意（节选）

```
你是小说续写助手。请依据以下信息续写第{chapter_number}章正文。

【全局规划】
{volume_summaries}

【本章大纲】
{chapter_outline}

【场景执行清单】
{scene_points}

【风格控制】描写密度：{density}/5，对话占比：{dialogue_ratio}/5，节奏：{pacing}

【需回收/呼应的活跃伏笔】
{active_foreshadowing_summary}

【最近时间线】
{recent_timeline}

【全部角色档案】
{all_characters_profiles}

【最新角色状态（上章末尾）】
{latest_snapshots}

【前文正文（最近15章全文）】
--- 第{N-15}章 ---
{content}

... (中间章节)

--- 第{N-1}章 ---
{content}

【早期相关片段】
(如有检索到：)
--- 关联第X章（关于{关键词}）---
{retrieved_paragraph}

请按场景要点执行，自然衔接上文，注意伏笔呼应。直接输出正文，末尾附元信息。
```

### 5.2 大纲生成 Prompt 调整

不再有独立的卷纲/阶段纲生成，总纲改为分卷概要。生成章纲时传入分卷概要和前文最近15章内容（或摘要）。

**章纲+场景要点生成 Prompt 示例**：
```
根据以下信息为第{chapter_number}章生成章纲和场景要点。

【分卷概要】
{volume_summaries}

【前文最近剧情摘要】（基于最近15章自动提炼）
{recent_summary}

【角色状态】
{latest_snapshots}

要求：
1. 输出本章核心目标（一句话），情感走向，章末钩子。
2. 将本章分解为3-5个场景，每个场景写出：任务、关键对话提示、氛围要点、所占篇幅比例建议。
输出JSON格式：
{
  "title": "...",
  "core_objective": "...",
  "emotional_arc": "...",
  "hooks": "...",
  "scenes": [
    { "order":1, "mission":"...", "dialogue_hint":"...", "atmosphere":"...", "ratio":0.3 }
  ]
}
```

### 5.3 元数据自动提取 Prompt（每次续写后自动运行）

**伏笔/时间线提取**：
```
分析以下新写章节，提取：
1. 新埋设的伏笔/悬念（描述、重要性）。
2. 揭示或推进的已有伏笔（请指明描述关键词，尽量匹配已记录条目）。
3. 故事时间推进情况（相对上一章的时间变化、重要时间节点）。
4. 角色状态变化：每个出场角色的位置、情绪、人际关系或能力变化（与上章快照对比）。

已记录伏笔列表：{active_foreshadowing}
上一章时间线：{last_timeline}
上一章角色快照：{last_snapshots}

新章节正文：
{chapter_content}

输出JSON：
{
  "new_foreshadowings": [...],
  "resolved_foreshadowings": [{"keyword":"...", "chapter":...}],
  "timeline_updates": {...},
  "character_snapshots": {...}
}
```

### 5.4 风格控制指令

不再直接注入大段风格描述，而是添加一行动态指令：
```
【文风控制】描写详细度:{density}/5, 对话密集度:{dialogue_ratio}/5, 叙事节奏:{pacing}。
```
在系统指令中说明尺度含义（例如密度5为电影化细节描写，1为简洁叙述），模型即可适应。

### 6.2 核心交互流程

1. **创建项目** → 填入分卷概要（可由AI根据简介生成），导入已有前文（逐章或全文）。
2. **初始化章纲**：选择起始章节，AI根据概要+已有前文生成章纲+场景要点，用户调整。
3. **续写操作**：
   - 点击"续写下一章"，自动弹出向导，预填章纲（若已有人工编写）或提示先生成。
   - 用户可微调风格滑块（密度/对话/节奏），然后点击"开始续写"。
   - 流式输出正文，完成后自动调用元数据提取，右侧弹出"新伏笔/时间线更新"小面板，用户一键确认或标记。
4. **编辑与重写**：正文框可直接修改，修改后可手动触发元数据重提取。
5. **大纲管理**：大纲视图展示所有章纲列表（表格），支持原地编辑标题、目标、钩子，点击展开场景要点。
6. **对话辅助**：仅用于自由提问，如"提几个让本章冲突更激烈的建议"，AI回复不直接改动数据，用户采纳后手动操作。

## 七、上下文窗口管理（修改为15章底线+检索增强）

### 7.1 管理原则

- **硬保证：最近15章全文（约3.75万字，假设每章2500字）必然在上下文中。**
- **全局信息**：分卷概要、当前章纲+场景、风格参数、活跃伏笔摘要、最近时间线、角色档案及最新快照。
- **早期章节**：仅当续写需要关联早期伏笔、特定角色旧事时，才通过关键词检索相关原文片段（每个片段200-500字），最多插入3-5个片段。这可由AI在生成前判断"需不需要回忆某些早期内容"，或基于规则（章纲或场景要点中若提到了早期角色/事件关键词，自动检索）。

### 7.2 Token预算示例（50万字长篇，续写第200章）

- 系统指令+全局信息：5,000 tokens
- 角色档案+快照：5,000 tokens
- 分卷概要+章纲场景：1,500 tokens
- **最近15章全文**：15×2500字 ≈ 37,500 汉字 ≈ 约25,000 tokens（中文约1.5字/token）
- 检索早期片段（最多5段×300字）：≈ 1,000 tokens
- 输出空间：3,000 tokens
- **合计约 40,000 tokens**，远低于1M窗口，成本极低，且上下文高度聚焦。

### 7.3 检索实现建议

- 对 `chapters` 表的 `content` 字段建立全文索引（SQLite FTS5），按 `project_id` 过滤。
- 检索关键词来自：当前章纲中的角色名、伏笔关键词、场景要点中提示的"呼应XX事件"。
- 检索结果排名取最相关的前3段。

## 十、总结：主要优化点

1. **大纲极简化**：总纲→分卷概要，章节规划→章纲+场景要点，消除卷/阶段独立维护。
2. **上下文聚焦**：底线15章全文，其余按需检索，成本剧降且关键连贯性不打折。
3. **自动化闭环**：伏笔、时间线、角色状态由AI在每次续写后自动提取更新，人只审核。
4. **风格参数化**：每次续写可微调描写密度/对话占比/节奏，简单高效。
5. **交互重心转移**：向导式续写成为主操作，自然语言对话退居辅助，减少误操作。
6. **数据一致性**：正文入SQLite，利用FTS5加速检索，单文件易备份。

该方案在保持原方案核心功能的同时，大幅降低了使用与维护的认知负荷，使创作者能更专注于故事本身。
